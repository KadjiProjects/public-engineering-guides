Azure Cloud Expertise — A Concept-Mastery Map

Not a command cheat-sheet. The goal is to understand every capability of Azure well enough to make decisions and explain trade-offs to any employer. Commands are included only as documentation.
Philosophy of this document. Expertise is no longer about memorising az syntax — an AI assistant can compose and run any command on demand. What an employer pays for is judgement: knowing which service solves which problem, what the trade-offs are, and how the pieces fit into a secure, reliable, cost-effective whole. So each section leads with concepts and decision signals, and lists a representative command purely for the record. Treat every pre block as “this is the shape of the action; the AI executes it.”
This map is the 30,000-ft view of a full curriculum. It gives you the whole platform in one sitting; the Azure Expert Curriculum in this folder zooms each section into a deep module with .NET code, decision frameworks, a hands-on lab, and self-checks. 1The cloud mental model

Everything else hangs off four ideas. Internalise these and the rest is detail.

  • Shared Responsibility. Azure secures the cloud of the platform; you secure what you put in it. The line moves with the service model.
  • The service-model ladder. IaaS (you manage OS upward) → PaaS (you manage app + data) → FaaS/serverless (you manage just code) → SaaS (you manage only configuration). Higher = less ops, less control.
  • Control plane vs data plane. Azure Resource Manager (ARM) is the control plane — it creates/configures resources. The data plane is the resource doing its job (a query hitting SQL, a blob upload). RBAC and APIs differ between them.
  • Everything is a resource with an ID. Every object has a unique ARM resource ID, lives in a resource group, is described declaratively, and is governed by policy, RBAC, locks and tags. Master this and Azure becomes uniform.
Concept that signals depth: Elasticity vs scalability vs resilience are different. Elasticity = scale out/in automatically with demand. Scalability = the ceiling you can reach. Resilience = surviving failure. Many candidates conflate them; an expert separates them and designs for each deliberately.

2Global infrastructure

  • Geography → Region → Availability Zone → Datacenter. A region is a set of datacenters; an Availability Zone (AZ) is one or more physically isolated datacenters within a region with independent power/cooling/network.
  • Region pairs. Most regions pair with another in the same geography for replication and sequential platform updates — the basis of geo-redundancy and data-residency compliance.
  • Edge & sovereignty. Edge zones, Front Door PoPs, and sovereign/government clouds matter for latency and regulated data.
Expert signal: Pick a region for three reasons together — latency to users, data-residency law, and service availability (not every service is in every region). Then design across ≥3 AZs for zone-redundant resilience without multi-region cost.
# For the record — list regions and which support availability zones
az account list-locations --query "[].{Region:name}" -o table

3Resource hierarchy & the control plane

ScopePurpose
Management GroupApply RBAC + Policy across many subscriptions (org-wide guardrails).
SubscriptionBilling & isolation boundary; holds quotas/limits.
Resource GroupLifecycle unit — deploy, secure, and delete related resources together.
ResourceThe individual service instance.

Cross-cutting controls applied at any scope: RBAC (who can do what), Azure Policy (what’s allowed), Tags (metadata/cost), Resource Locks (prevent accidental delete), Azure Resource Graph (query the whole estate with KQL).

Inheritance: RBAC and Policy flow downward — a role or policy at a management group applies to every subscription/RG/resource beneath it. Designing the tree well is governance leverage.
# Query every public-IP across the whole estate in one shot
az graph query -q "Resources | where type=='microsoft.network/publicipaddresses'"

4Identity & access — Microsoft Entra

In the cloud, identity is the perimeter. This is the most important security domain.

  • Microsoft Entra ID — the identity provider: users, groups, app registrations, service principals, SSO, MFA.
  • RBAC — role assignments (scope + role + principal). Built-in roles (Owner/Contributor/Reader) plus custom roles. Least privilege is the rule.
  • Managed Identity — an Entra identity for a resource so it calls other services without storing secrets. System-assigned (tied to one resource) vs user-assigned (shared). The modern default.
  • Conditional Access — policy engine: “require MFA from untrusted networks”, “block legacy auth”. The Zero-Trust enforcement point.
  • Privileged Identity Management (PIM) — just-in-time, time-boxed elevation for admin roles with approval/audit.
  • Entra External ID / B2B / B2C — collaborate with partners and authenticate customers.
Expert signal: “No secrets in code or config.” App gets a managed identity → assigned an RBAC role on SQL/Storage/Key Vault → authenticates with a token, never a password. Combine with PIM for admins and Conditional Access for users and you’ve described a Zero-Trust identity posture.
# Give a web app a managed identity and grant it read on a Key Vault
az webapp identity assign -g rg-app -n my-app
az role assignment create --assignee <app-principal> \
   --role "Key Vault Secrets User" --scope <keyvault-id>

5The compute spectrum

Know the full ladder and the one-line reason to choose each:

ServiceChoose when
Virtual MachinesFull OS control / lift-and-shift / specialised software.
VM Scale Sets (VMSS)Identical VMs that autoscale behind a load balancer.
App ServiceManaged web apps/APIs — slots, autoscale, SSL, no servers.
Azure FunctionsEvent-driven/serverless; pay-per-execution; scale to zero.
Container AppsServerless containers + microservices (Dapr, KEDA scaling) without managing K8s.
Azure Kubernetes Service (AKS)Full Kubernetes for complex, portable, multi-service platforms.

The serverless idea: you stop paying for idle. Functions and Container Apps can scale to zero and bill per request/second — ideal for spiky or unpredictable load, the opposite of a reserved VM you pay for 24/7.
.NET Functions clock-facts (verified Jul 2026): the in-process model's support ends 2026-11-10 — the isolated worker model is the only .NET path forward (and the only one supporting .NET 10 and Flex). Flex Consumption is now the recommended serverless plan: VNet integration, per-function scaling to ~1000 instances, always-ready instances to defeat cold starts. Functions runtime 1.x retires 2026-09-14. Details: Module 04.
# A serverless container service that scales 0→N on HTTP load
az containerapp create -g rg -n api --image myacr.azurecr.io/api:1.0 \
   --min-replicas 0 --max-replicas 20 --ingress external --target-port 8080

6Containers & orchestration

  • Azure Container Registry (ACR) — private image registry; supports tasks, geo-replication, and content trust.
  • AKS — managed Kubernetes control plane (free) + your worker node pools. You own deployments, services, ingress, scaling (HPA/KEDA), and upgrades.
  • Container Apps — opinionated, serverless abstraction over Kubernetes/KEDA; great when you want containers without cluster ops.
  • Dapr — sidecar runtime for service-to-service calls, state, and pub/sub, portable across Container Apps and AKS.
Expert signal: Be able to say when not to use Kubernetes. AKS is powerful but it’s an operational commitment (upgrades, node patching, networking, RBAC). For a handful of services, Container Apps or App Service deliver 80% of the value with a fraction of the ops burden.
az aks create -g rg -n cluster --node-count 3 --enable-managed-identity --generate-ssh-keys
az aks get-credentials -g rg -n cluster   # then kubectl, via AI

7Networking — the expert differentiator

Strong networking knowledge is what separates a senior cloud engineer from an app developer who deploys.

  • VNet & Subnets — private address space; segment tiers and services.
  • NSG / Application Security Groups — stateful firewall rules; ASGs let you write rules against app roles instead of IPs.
  • VNet Peering & Hub-Spoke — connect VNets; centralise shared services (firewall, DNS, gateways) in a hub.
  • Load balancing ladder — Load Balancer (L4 regional) → Application Gateway (L7 + WAF, regional) → Front Door (global L7 + CDN + WAF) → Traffic Manager (DNS-based global routing).
  • Private Link / Private Endpoint — bring PaaS services (SQL, Storage) into your VNet with a private IP; remove public exposure.
  • Azure Firewall & DDoS Protection — managed, scalable network security; central egress control.
  • Azure Bastion — browser-based RDP/SSH with no public VM IPs.
  • Hybrid — VPN Gateway (encrypted over internet) vs ExpressRoute (private dedicated circuit).
  • DNS — public Azure DNS + Private DNS Zones (essential for private endpoints).
The hub-spoke pattern is the reference enterprise topology: a central hub VNet holds shared firewall, DNS and the on-prem gateway; spoke VNets hold workloads and peer to the hub. Know this diagram cold.
# Make a SQL server reachable only privately
az network private-endpoint create -g rg -n pe-sql --vnet-name vnet --subnet data \
   --private-connection-resource-id <sql-id> --group-id sqlServer --connection-name c1

8Storage

  • Storage Account bundles four services: Blob (objects), File (SMB/NFS shares), Queue (basic messaging), Table (key-value NoSQL).
  • Blob access tiers — Hot / Cool / Cold / Archive trade price for retrieval latency; the newer Smart tier auto-moves blobs between hot/cool/cold by usage, and lifecycle policies move data automatically.
  • Data Lake Storage Gen2 — Blob + hierarchical namespace for analytics.
  • Managed Disks — VM block storage (Standard HDD → Premium/Ultra SSD).
  • Redundancy — LRS (3 copies, 1 DC) → ZRS (across zones) → GRS/GZRS (across regions). Pick by RPO and compliance.
  • Access — keys, SAS tokens, or (best) Entra + managed identity; encryption at rest is on by default.
Expert signal: Match redundancy tier to the recovery objective and access tier to the data’s temperature — then automate tier transitions with lifecycle rules. That’s cost-aware design, not just “put it in Blob.”

9Databases

ServiceBest for
Azure SQL DatabaseManaged single relational DB; serverless & elastic pools; hyperscale.
SQL Managed InstanceNear-full SQL Server surface for lift & shift.
Cosmos DBGlobal, multi-model, single-digit-ms NoSQL; choose the consistency level.
Azure DB for PostgreSQL / MySQLManaged open-source relational.
Azure Cache for RedisIn-memory cache / session / pub-sub.
Synapse / Fabric WarehouseAnalytical (OLAP) workloads at scale.
Cosmos DB consistency spectrum — Strong → Bounded Staleness → Session → Consistent Prefix → Eventual. It’s the cleanest real-world illustration of the CAP / PACELC theorem (Microsoft’s own docs cite PACELC): you trade consistency for latency/availability, explicitly. Being able to explain this is a senior marker.
OLTP vs OLAP — transactional databases (SQL, Cosmos) serve the app; analytical stores (Synapse, Fabric, Data Lake) serve reporting/ML. Don’t run heavy analytics on your OLTP store — separate them.

10Data & analytics

  • Microsoft Fabric — the unified SaaS analytics platform (OneLake lake, warehouse, Data Factory, Power BI) — the strategic direction for analytics.
  • Azure Synapse Analytics — integrated big-data + warehousing (SQL pools, Spark).
  • Azure Data Factory — managed ETL/ELT orchestration and data movement.
  • Azure Databricks — Spark-based analytics & ML at scale.
  • Event Hubs — high-throughput streaming ingestion (millions of events/sec).
  • Stream Analytics — real-time SQL-like processing over streams.
  • Data Lake — the cheap, scalable landing zone for raw data.
Expert signal: Sketch the canonical pipeline — ingest (Event Hubs/Data Factory) → store (Data Lake/OneLake) → process (Databricks/Synapse/Stream Analytics) → serve (Warehouse/Power BI) — and name where governance and lineage live. Architecture over tooling.

11Messaging & integration

Knowing which messaging primitive to use is a classic interview discriminator:

ServiceUse for
Service BusEnterprise messaging — queues & topics, ordering, transactions, dead-letter. Commands/work.
Event GridReactive, discrete event notifications (pub/sub) with near-real-time delivery.
Event HubsHigh-volume telemetry/event streaming & analytics ingestion.
Storage QueuesSimple, cheap, large queues without enterprise features.
Logic AppsLow-code workflow/integration across 1000+ connectors.
API Management (APIM)Gateway for your APIs — auth, throttling, versioning, a developer portal.
The one-liner that proves you get it: “Event = something happened (Event Grid/Hubs); Message = please do something (Service Bus). Events are facts; messages carry intent.” Plus: messaging enables loose coupling and resilience — producers and consumers scale and fail independently.

12AI & machine learning

  • Microsoft Foundry (the rebrand of Azure AI Studio / Azure AI Foundry) — the unified PaaS for building, deploying and governing generative-AI apps and agents; Azure OpenAI is now a Foundry resource, and the old Azure AI Services are surfaced as “Foundry Tools.” One resource, unified RBAC/networking/policy, built-in tracing & evaluations.
  • Azure OpenAI (in Foundry) — enterprise-governed access to frontier LLMs (GPT-4o, o-series, etc.) with private networking and content filtering.
  • Foundry Tools / Azure AI Services — pre-built APIs: vision, speech, language, document intelligence, translation.
  • Azure AI Search — vector + keyword retrieval; the backbone of RAG (retrieval-augmented generation).
  • Azure Machine Learning — full MLOps: train, register, deploy, monitor custom models.
Expert signal: Describe a governed RAG pattern on Azure — data in Blob/Data Lake → embeddings indexed in AI Search → an Azure OpenAI model grounded on retrieved chunks → exposed through APIM with managed-identity auth and Application Insights tracing. That single answer ties AI to security, networking, and observability at once.

13Security & Zero Trust

  • Zero Trust — “never trust, always verify.” Verify explicitly, least privilege, assume breach. The governing mindset.
  • Microsoft Defender for Cloud — posture management (Secure Score) + workload threat protection (CSPM/CWPP).
  • Microsoft Sentinel — cloud-native SIEM/SOAR for detection, hunting, and automated response.
  • Key Vault — secrets, keys, certificates; with HSM-backed tiers and rotation.
  • Encryption — at rest (platform-managed or customer-managed keys) and in transit (TLS); confidential computing for in-use.
  • Network security — WAF (on App Gateway/Front Door), DDoS Protection, Azure Firewall, Private Link to remove public surface.
Defense in depth — layer controls (identity → network → host → app → data) so no single failure is fatal. Pair with least privilege and assume-breach and you can articulate any security review.

14Governance & cost management

  • Azure Policy — enforce/audit rules (allowed regions, required tags, enforced encryption); deploy-if-not-exists remediation.
  • Landing Zones — governed, repeatable foundations (the CAF “Ready” phase) workloads deploy into.
  • Cost Management + Budgets — analyse, allocate (by tag), and alert on spend.
  • Azure Advisor — automated recommendations across cost, security, reliability, performance.
  • Pricing levers — Pay-as-you-go, Reservations, Savings Plans, Spot, and Azure Hybrid Benefit (reuse Windows/SQL licences).
Expert signal — FinOps: cost is a continuous engineering discipline, not a monthly surprise. “Right-size from telemetry, reserve the steady-state baseline, burst on consumption, apply Hybrid Benefit, and allocate every cost with tags” is a complete cost-governance answer.
az policy assignment create --name 'require-tag-env' \
   --policy <require-tag-policy-id> --scope <mg-or-sub-id>

15Infrastructure as Code & DevOps

  • Bicep — Azure-native declarative IaC (compiles to ARM). The first language to master.
  • ARM templates — the underlying JSON; read, rarely hand-write.
  • Terraform — multi-cloud declarative IaC; common in larger orgs.
  • CI/CD — GitHub Actions / Azure DevOps Pipelines build, test, and deploy both app and infra — authenticated via workload identity federation (OIDC), so pipelines hold zero stored credentials.
  • Patterns — environments (dev→test→prod), App Service deployment slots / Container Apps revision weights for blue-green & canary, feature flags (App Configuration) to decouple release from deploy, policy-gated deployments, deployment stacks for drift-denial and clean teardown.
  • Aspire + azd — for .NET distributed apps: Aspire models the system in C# (local orchestration, dashboard, OTel defaults); azd up provisions and deploys it to Container Apps or App Service. The fastest laptop→Azure path that still yields sane infrastructure.
Declarative vs imperative — IaC describes the desired state; the engine reconciles reality to it (idempotent). That’s why portal-clicking doesn’t scale and IaC does: it’s reviewable, repeatable, and auditable.
# Deploy a whole environment from a declarative file (AI runs it)
az deployment group create -g rg --template-file main.bicep --parameters env=prod

16Observability

  • Azure Monitor — the umbrella for metrics, logs, traces, alerts.
  • Application Insights — APM: requests, dependencies, exceptions, distributed tracing, live metrics.
  • Log Analytics + KQL — central log store queried with Kusto Query Language (learn KQL — it’s the lingua franca of Azure ops).
  • Alerts + Action Groups — condition → notify/automate/auto-heal.
  • Workbooks & Dashboards — shared operational and exec views.
The three pillars of observability — metrics (what), logs (why), traces (where) — let you answer “is it healthy, and if not, where’s the fault?” Tie this to the SLA: you cannot promise availability you cannot measure.

17Reliability & disaster recovery

  • Availability Zones for datacenter-fault tolerance; multi-region for region-fault tolerance.
  • RTO / RPO — the two numbers that define every DR design; ask the business for them first.
  • Composite SLA — chaining services multiplies their SLAs (two 99.9% services in series ≈ 99.8%). Redundancy raises it back. Be able to do this maths.
  • Azure Site Recovery / Backup — VM failover and point-in-time restore.
  • Patterns — active-active vs active-passive; retry with backoff, circuit breaker, bulkhead, health probes.
  • Chaos Studio — fault injection to prove resilience.
Trade-off to verbalise: every additional “nine” multiplies cost and complexity. Design to the business’s stated SLA — no more, no less. Over-engineering resilience is as much a failure as under-engineering it.

18The two frameworks that tie it together

Well-Architected Framework (WAF) — judge any single workload across five pillars: Reliability, Security, Cost Optimization, Operational Excellence, Performance Efficiency.

Cloud Adoption Framework (CAF) — the org-level journey: Strategy → Plan → Ready → Adopt → Govern → Manage.

Expert signal: Use WAF as a checklist for every design review and CAF as the narrative for every transformation. Naming the right framework for the right altitude (one workload vs the whole org) instantly reads as senior.

19Proving expertise to an employer

  • Certifications as proof-of-breadth — AZ-900 (fundamentals) → AZ-104 (administrator) → role specialisations: AZ-204 (developer), AZ-305 (solutions architect — the gold standard for design; it has AZ-104 as a prerequisite and a 700 pass score), AZ-500 (security), AZ-700 (networking), DP-203 (data), AI-102 (AI engineer). AZ-305 is the one that signals architecture judgement.
  • Speak in trade-offs, not features. “I’d use X because Y, accepting trade-off Z, mitigated by W.” Every section above is written to feed that sentence.
  • Always close the loop on five things in any design: identity (managed identity, least privilege), networking (private, segmented), reliability (zones, SLA, DR), observability (monitor + alerts), and cost (right-size, reserve, tag).
  • Have one reference architecture ready — e.g. a web app: App Service (zone-redundant) → SQL (private endpoint) → Key Vault + managed identity → Front Door + WAF → App Insights → Bicep + CI/CD → reserved + Hybrid Benefit. If you can defend one end-to-end design across all five pillars, you can defend any.
The mindset, in one line: An Azure expert isn’t someone who remembers commands — it’s someone who, given a business problem and its constraints, can choose the right services, justify the trade-offs across the five pillars, and hand the exact execution to automation (or an AI) with confidence that the design is sound.
