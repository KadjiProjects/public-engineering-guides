Azure Expert Curriculum

For an experienced .NET developer becoming an expert at developing, deploying, and configuring Azure services. Everything in this folder is one system: deep modules to build understanding, hands-on labs to build skill, and drill/recall documents to make it permanent.
What "expert" means here — and how this curriculum gets you there.
An Azure expert is someone who, given a business problem and its constraints, chooses the right services, justifies the trade-offs across reliability, security, cost, operations and performance, builds it with production-grade code and IaC, and can prove the system is healthy. That is four capabilities — judgment, depth, hands-on skill, and recall — and each document type in this folder trains exactly one of them.

1 · How to learn from this folder (read this first)

This curriculum uses four evidence-based techniques. They are not optional decoration — they are why the material is organized the way it is:

TechniqueWhat it isWhere it lives here
Active recallRetrieving from memory beats re-reading, by a wide margin. Test yourself before you feel ready.Flashcards after every module; self-check questions at each module's end.
Spaced repetitionReviewing at growing intervals (1 → 3 → 7 → 21 days) moves knowledge to long-term memory.The weekly plan below schedules reviews; the flashcard app tracks "knew it / review again".
Deliberate practiceSkill comes from doing slightly-too-hard things with feedback, not from watching or reading.The lab track — one capstone application built progressively across all 12 modules.
Elaboration (Feynman)Explaining a concept in your own words exposes what you don't actually understand.Each module ends with "explain out loud" prompts; the assessment has design exercises with model answers.
The weekly rhythm (repeat for each module): Day 1–2 — read the module actively (pen, not highlighter: write questions in margins). Day 3–4 — do the module's lab section hands-on in a real Azure subscription. Day 5 — drill the flashcards for this module + all previous modules marked "review again." Day 6 — answer the module's self-check from memory, then do one "explain out loud" prompt as if teaching a colleague. Day 7 — rest, or re-do whatever failed.
Non-negotiable: you need a real Azure subscription (free tier + ~$20–50 of experiments is enough — every lab includes cleanup steps and cost guards). Reading about deployment slots is knowledge; swapping one under live traffic is expertise. The gap between those two is exactly what this curriculum closes.

2 · The 12-module path (≈12 weeks at 6–8 h/week)

Week 1

01 · Foundations & the Azure resource model

Mental models, ARM, hierarchy, governance primitives, the CLI/tooling stack — the grammar everything else is written in.

Week 2

02 · Identity, access & secrets

Entra ID, RBAC, managed identity, Key Vault — with the exact .NET code (DefaultAzureCredential) that makes "no secrets" real.

Week 3

03 · App Service deep dive

Plans & SKUs, slots, configuration, scaling, networking integration, deployment — the .NET workhorse, mastered.

Week 4

04 · Functions & serverless

Isolated worker model, triggers/bindings, hosting plans (Flex Consumption!), Durable Functions, cold starts.

Week 5

05 · Containers: ACR → Container Apps → AKS

Images, registries, Container Apps (revisions, KEDA, Dapr), when AKS, and Aspire for local orchestration.

Week 6

06 · Relational data & caching

Azure SQL tiers & purchasing models, EF Core in the cloud, connection resilience, Redis and cache patterns.

Week 7

07 · Cosmos DB & storage

Partitioning, RU/s, consistency levels in practice, the .NET SDK done right; Blob storage patterns and SAS vs identity.

Week 8

08 · Messaging & eventing

Service Bus (sessions, DLQ, dedup), Event Grid, Event Hubs — and the .NET consumer patterns that survive production.

Week 9

09 · Networking

VNets, private endpoints + the DNS mechanics everyone gets wrong, the load-balancing ladder, hub-spoke, App Service/Functions network integration.

Week 10

10 · Observability

App Insights + OpenTelemetry in .NET, a working KQL tutorial, alerts, dashboards, SLOs — prove health, don't assert it.

Week 11

11 · IaC & delivery

Bicep from zero to modules, GitHub Actions with OIDC (no deploy secrets), environments, blue-green/canary, azd.

Week 12

12 · Architecture, reliability & cost

Well-Architected in practice, resilience patterns, DR & SLA math, FinOps, and defending a reference architecture end-to-end.

5 · Keeping it current

Azure changes monthly; this folder is verified against Microsoft Learn as of July 2026. Every module ends with a short "verify before you build" list of the facts most likely to drift (SKUs, limits, previews, retirements). Two standing rules:

  • Before committing to any design, check exact limits/pricing on Microsoft Learn and Azure Updates.
  • Known clock-driven facts baked into this material: Functions in-process model support ends 2026-11-10 (isolated worker is the only .NET path forward); Functions runtime 1.x ends 2026-09-14; Flex Consumption is the recommended serverless plan; Aspire is the local-orchestration + deployment story for .NET distributed apps; the AI platform is Microsoft Foundry (Azure OpenAI is a Foundry resource).
