Azure Migration & Deployment — A Knowledge Map

For a senior .NET developer assessing whether a legacy system can move to Azure, and how to deploy it. Each section: what it is, why it matters for a migration decision, and how to demonstrate the skill. Part of the Azure Expert Curriculum — read this alongside Module 12, as the "apply everything to a legacy estate" scenario.

You already know how reliable software gets built. The job here is mapping that judgement onto a new vocabulary. This document is organised the way a real assessment flows: understand the frameworks → inventory the old system → choose targets → design the platform (compute, data, network, identity) → govern, automate, observe, and cost it → execute the migration.

1The two frameworks you must name-drop correctly

Microsoft structures everything around two complementary frameworks. Knowing the difference is the single fastest way to sound credible.

Cloud Adoption Framework (CAF) — the journey

The end-to-end methodology for an organisation moving to cloud. Phases: Strategy → Plan → Ready (landing zone) → Adopt (migrate/innovate) → Govern → Manage. CAF answers “how do we as a company do this safely and repeatably.”

Well-Architected Framework (WAF) — the quality of one workload

Five pillars to judge any single solution. Memorise them:

PillarThe question it asks
ReliabilityDoes it recover from failure and meet its SLA?
SecurityIs data and access protected, least-privilege, encrypted?
Cost OptimizationAre we paying only for value delivered?
Operational ExcellenceCan we deploy, monitor and operate it with automation?
Performance EfficiencyDoes it scale to demand without waste?
How to demonstrate it: When proposing any target architecture, narrate the trade-off across these five pillars. “I’d put it on App Service for operational excellence and cost, accept a small reliability trade-off, and mitigate with zone-redundancy.” That sentence shape signals seniority.

2Migration strategies — the “6 R’s”

Every legacy component is sorted into one of these. The whole assessment is essentially assigning an R to each workload.

StrategyWhat it meansEffort / reward
Rehost (“lift & shift”)Move the VM/app as-is to Azure VMs.Lowest effort, least cloud benefit. Good first step.
Replatform (“lift & reshape”)Minor changes — e.g. app onto App Service, DB onto Azure SQL Managed Instance.Moderate effort, removes server management.
RefactorRecode parts to fit PaaS (containerise, swap a library, add config).More effort, better scaling/cost.
RearchitectChange the structure — monolith to services, add queues, go event-driven.High effort, high long-term payoff.
RebuildRewrite from scratch as cloud-native.Highest effort; only when the old code is a liability.
ReplaceDrop it for SaaS (e.g. swap a custom CRM for a product).No code; a procurement decision.

Two bonus R’s in the inventory phase: Retain (leave on-prem for now) and Retire (switch it off — you’ll be surprised how much you can).

Senior judgement: The trap is rewriting everything. The right answer for an old system is usually rehost or replatform first to stop the bleeding, then refactor incrementally. State that bias explicitly — it shows you optimise for risk and time-to-value, not novelty.

3Assessing the legacy system — what to inventory

Before choosing a target you characterise the source. These are the dimensions that actually drive the decision:

  • Runtime & framework version — .NET Framework 4.x vs modern .NET (8/9/10)? Framework runs only on Windows; that constrains you to Windows App Service / Windows containers / VMs, or signals a refactor to cross-platform .NET. (For Functions targets: the isolated worker model is mandatory going forward — in-process support ends 2026-11-10.)
  • OS & its support status — Windows Server 2012/2016 out of support? That alone can justify the move (and Azure offers Extended Security Updates).
  • State & session — Does it hold in-memory session or write to local disk? Cloud instances are disposable; stateful assumptions break horizontal scaling. (See the .NET note below.)
  • Dependencies & coupling — COM components, Windows services, scheduled tasks, registry use, file-share paths, hard-coded server names, SMTP relays.
  • Integrations — what calls it and what it calls (other systems, third-party APIs, on-prem databases). These dictate networking and cutover order.
  • Database — engine, version, size, features used (CLR, SQL Agent jobs, linked servers, cross-db queries). This decides Azure SQL DB vs Managed Instance vs SQL-on-VM.
  • Authentication — Windows/AD auth, custom membership, certificates. Maps to Entra ID / managed identity work.
  • Non-functional needs — throughput, latency, RTO/RPO, compliance/data residency, peak patterns.
  • Licensing — Windows/SQL licences you can carry with Azure Hybrid Benefit.
.NET analogy — the “12-factor” shift: On-prem you treated a server as a pet — it had a name, local state, and lived for years. In Azure you treat compute as cattle: stateless, replaceable instances behind a load balancer. So session state moves out (to Redis / SQL / cookies), config moves to App Settings & Key Vault (not web.config on disk), logs go to a sink (not local files), and uploads go to Blob storage (not the local file system). Spotting these “pet” assumptions in the old code is the assessment.

4Discovery & assessment tooling

You don’t do this by hand. Name these tools and what each produces:

ToolWhat it does
Azure MigrateThe hub. Discovers on-prem servers (agentless appliance), maps dependencies, right-sizes Azure targets, and estimates cost. Your primary assessment artefact.
Data Migration Assistant (DMA)Scans SQL Server for compatibility issues and feature blockers against Azure SQL targets.
Azure Database Migration Service (DMS)Performs the actual data move, including near-zero-downtime online migrations.
App Service Migration AssistantChecks an IIS/.NET site for App Service readiness and can move it.
.NET Upgrade Assistant / try-convertHelps move .NET Framework projects toward modern .NET.
TCO / Pricing CalculatorBuilds the business case — on-prem cost vs projected Azure cost.
How to demonstrate it: Say you’d start with an Azure Migrate dependency map before drawing any target architecture — because you migrate in dependency order and never strand a chatty integration across the internet. That’s the line that separates an assessor from a guesser.

5Azure fundamentals & the resource hierarchy

The organising containers, top to bottom — this is the cloud equivalent of solution → project → class structure:

  • Tenant (Entra ID directory) — your organisation’s identity boundary.
  • Management Group — groups subscriptions to apply policy/RBAC at scale.
  • Subscription — billing + isolation boundary; quotas live here.
  • Resource Group — a lifecycle bucket for resources deployed/deleted together (think one app environment).
  • Resource — the actual thing (a web app, a database, a storage account).
  • Region & Availability Zone — where it physically runs; a region has multiple zones (independent datacentres) for resilience.
.NET analogy: Resource Group ≈ a .sln grouping related projects you build and ship together; Region ≈ which datacentre your bin actually runs in; Tags ≈ assembly attributes/metadata you query later (for cost and ownership).

6Compute / hosting options core decision

The heart of the migration choice. Ordered from “most control / most work” to “most managed”:

ServiceUse it when….NET fit
Virtual Machines (IaaS)Pure lift & shift; you need OS control or have un-containerisable dependencies.Any app, incl. .NET Framework. You still patch the OS.
App Service (PaaS)A web app or API you want hosted with zero server management, easy scaling, slots, custom domains/SSL.The natural home for an ASP.NET / ASP.NET Core site. Supports Windows for Framework apps.
Azure FunctionsEvent-driven / scheduled / bursty work; pay-per-execution (serverless).Replaces Windows services, cron jobs, queue workers.
Container AppsMicroservices/containers with built-in scaling (incl. scale-to-zero) without running Kubernetes yourself.Great middle ground for containerised .NET services.
AKS (Kubernetes)Many services, complex orchestration, and a team that can run K8s.Powerful but operationally heavy — don’t pick it to host one app.
Service FabricLegacy stateful microservice platform (older Microsoft estates).Mostly maintenance scenarios now.
Decision discipline: Default to the most managed option the workload allows. Move down to VMs only when a real constraint forces it. Justify each step down — that restraint is what reviewers look for.
.NET analogy: App Service is to a VM what a hosted/managed runtime is to “I install IIS, configure the app pool, and patch Windows myself.” Functions is the cloud version of a BackgroundService/Windows Service that only wakes when there’s work — and you’re billed only while it runs.

7Data & storage

Relational (your SQL Server almost certainly lands here)

TargetPick when…
Azure SQL DatabaseA single modern DB; you want fully managed, auto-patched, serverless/elastic options. No SQL Agent, limited instance-level features.
Azure SQL Managed InstanceLift & shift of a real SQL Server — keeps SQL Agent, cross-DB queries, CLR, near-full surface. Best for legacy.
SQL Server on a VMYou need OS/instance-level control or an unsupported feature. Most work, most control.
Azure DB for PostgreSQL / MySQLOpen-source relational workloads.

Non-relational & storage primitives

  • Azure Storage Account — four services in one: Blob (files/objects, the workhorse), Queue (simple messaging), Table (cheap key-value), File (SMB share — handy to replace an on-prem file share with minimal code change).
  • Azure Cosmos DB — globally distributed, low-latency NoSQL; pick for planet-scale or flexible schema, not as a default.
  • Azure Cache for Redis — externalises session/cache state so your compute can stay stateless.
.NET analogy: Moving local-disk file writes to Blob is the same refactor as swapping File.WriteAllBytes for a storage SDK call. Replacing in-proc IMemoryCache shared across instances with Redis (IDistributedCache) is the canonical “make it scale horizontally” change.

8Networking

The part most app developers under-rate and most assessors get quizzed on.

  • Virtual Network (VNet) & Subnets — your private network in Azure; segment tiers (web/app/data) into subnets.
  • Network Security Group (NSG) — stateful firewall rules on subnets/NICs (allow 443, deny the rest). Think IP-table/firewall as config.
  • Load Balancer (L4) vs Application Gateway (L7, with WAF) vs Front Door (global L7 + CDN + WAF) — distribute traffic at increasing levels of smarts and reach.
  • Private Endpoint / Service Endpoint — keep traffic to PaaS (SQL, Storage) on the Azure backbone instead of the public internet. Expect to be asked “how do you make the database not publicly reachable?” — the answer is a Private Endpoint.
  • Hybrid connectivity — VPN Gateway (encrypted over internet) or ExpressRoute (private dedicated line) to reach on-prem during a phased migration.
  • Azure DNS / Private DNS Zones — name resolution, including for private endpoints.
Migration reality: During a phased move, the app is in Azure but the database may still be on-prem (or vice-versa). The networking question — latency, bandwidth, and a secure hybrid link — is usually the hardest practical constraint. Raise it early.

9Identity & security

  • Microsoft Entra ID (formerly Azure AD) — the cloud identity provider; users, groups, app registrations, SSO, MFA, conditional access.
  • RBAC (Role-Based Access Control) — who can do what on which resource (Owner/Contributor/Reader + granular roles), assigned at any level of the hierarchy.
  • Managed Identity — an Entra identity for your app so it authenticates to SQL/Key Vault/Storage with no secrets in code or config. This is the modern best practice — emphasise it.
  • Azure Key Vault — central store for secrets, keys, certificates; your app reads them at runtime via managed identity.
  • Microsoft Defender for Cloud — posture management + threat protection, with a “secure score” to track hardening.
  • Encryption — at rest (default on storage/SQL) and in transit (TLS); know that both are expected baselines.
.NET analogy: The old connectionString with a SQL username/password in web.config is exactly what you eliminate. The pattern becomes: app has a Managed Identity → granted an RBAC role on the DB → connects with no password at all. Secrets that must exist live in Key Vault, referenced — never committed.

10Governance & landing zones

  • Landing Zone — a pre-built, governed Azure foundation (networking, identity, policy, management) you deploy before migrating workloads. The CAF “Ready” phase. Workloads land into it.
  • Azure Policy — rules enforced automatically (“deny resources outside West Europe”, “require encryption”, “enforce tagging”). Compliance as code.
  • Tags — key/value metadata on resources for cost allocation, ownership, environment. Trivial to skip, painful to retrofit.
  • Blueprints / template specs — repeatable, governed environment definitions.
  • Cost & quota boundaries — subscriptions and management groups used deliberately to separate prod/non-prod and business units.
How to demonstrate it: Mention that you’d migrate into a landing zone rather than hand-crafting resources, so security and cost controls exist from day one. It signals you think at organisation scale, not just one app.

11Infrastructure as Code & CI/CD

  • Bicep — Azure-native, readable IaC language (the modern replacement for raw ARM JSON). Learn this one first.
  • ARM templates — the underlying JSON; you’ll read it, rarely write it by hand now.
  • Terraform — multi-cloud IaC; common in larger orgs.
  • Azure DevOps (Repos/Pipelines/Boards) & GitHub Actions — build/test/deploy pipelines. Either can deploy Bicep and push app code.
  • Deployment patterns — App Service deployment slots for blue-green/zero-downtime swaps; environments promoted dev → test → prod.
.NET analogy: IaC is dotnet build for your infrastructure — declarative, version-controlled, repeatable, code-reviewed. “It works because someone clicked the portal” is the cloud equivalent of an undocumented manual build step; you replace it with a pipeline.

12Observability

  • Azure Monitor — the umbrella for metrics, logs, and alerts.
  • Application Insights — APM for your app: request rates, dependencies, exceptions, end-to-end traces, live metrics. Drop-in for .NET.
  • Log Analytics (KQL) — the query workspace where logs land; Kusto Query Language is the SQL-like language you’ll use to investigate.
  • Alerts & Action Groups — fire on a metric/log condition → email/Teams/webhook/auto-heal.
  • Dashboards & Workbooks — shared operational views.
How to demonstrate it: Tie observability back to the WAF “Operational Excellence” pillar and to your SLA: “I can’t claim 99.9% if I can’t see failures — App Insights + alerts is part of the migration, not an afterthought.”

13Reliability & disaster recovery

  • SLA & the “number of nines” — know that higher availability needs redundancy you must design in (and pay for).
  • Availability Zones — spread instances across physically separate datacentres in one region to survive a datacentre fault.
  • Region pairs & multi-region — for surviving a whole-region outage; usually active-passive for cost.
  • RTO / RPO — Recovery Time / Recovery Point Objectives — the two numbers that define your DR design. Always ask the business for them.
  • Backup & geo-replication — Azure Backup, SQL automated backups + geo-restore, Storage redundancy tiers (LRS/ZRS/GRS), Azure Site Recovery for VM failover.
  • Scaling — vertical (bigger SKU) vs horizontal (more instances); autoscale rules on metrics.
Cost ↔ resilience trade-off: Every nine costs money. The senior move is to ask for the business’s RTO/RPO and SLA target first, then design the minimum architecture that meets it — not to gold-plate by reflex.

14Cost management

  • Pricing models — Pay-as-you-go, Reserved Instances (1/3-yr commit, big discount), Savings Plans (flexible commit), and Spot (cheap, interruptible).
  • Azure Hybrid Benefit — reuse existing Windows Server / SQL licences to cut compute cost dramatically. Always check this for a migration from a licensed estate.
  • Consumption vs provisioned — serverless (Functions, serverless SQL) bills per use; provisioned bills for reserved capacity. Match to traffic shape.
  • Microsoft Cost Management + Budgets — track spend, set budgets/alerts, analyse by tag.
  • TCO Calculator — builds the on-prem-vs-cloud business case (include hardware, power, ops labour, not just VM price).
How to demonstrate it: Cost is a first-class design input, not an afterthought. “Right-size from the Azure Migrate report, apply Hybrid Benefit, reserve the steady-state baseline, and let bursty load run on consumption” is a complete, credible cost story.

15Executing the migration

  1. Discover & assess — inventory + dependency map + right-size + business case (sections 3–4).
  2. Plan waves — group workloads by dependency; migrate in order; nothing chatty left stranded across the link.
  3. Build the landing zone — networking, identity, policy, monitoring foundation.
  4. Pilot — migrate one low-risk workload end-to-end; prove the runbook.
  5. Migrate data — DMS online migration to keep downtime minimal; validate row counts and integrity.
  6. Cutover — DNS/traffic switch in a planned window, with a tested rollback path.
  7. Validate — functional, performance, and security tests against the migrated system.
  8. Optimise (post-migration) — right-size again on real telemetry, then refactor toward PaaS where it pays off.
Always have a rollback. A migration without a tested back-out plan is a gamble. Stating that you design cutover with a rollback window is, by itself, a strong seniority signal.

16A decision cheat-sheet

Map a legacy trait straight to an Azure choice:

If the old system…Lean toward…
is .NET Framework, Windows-only, hard dependenciesRehost (VM) or App Service for Windows; refactor later.
is a standard ASP.NET / Core web app or APIApp Service (+ deployment slots).
has Windows services / scheduled jobs / queue workersAzure Functions or Container Apps.
is a real SQL Server using Agent/CLR/cross-DBAzure SQL Managed Instance.
is one clean modern databaseAzure SQL Database (serverless if bursty).
writes files to local diskBlob (objects) or Azure Files (SMB share).
keeps in-memory session across requestsExternalise to Redis / SQL; make compute stateless.
stores secrets in configKey Vault + Managed Identity (no secrets in code).
must talk to on-prem during transitionVPN Gateway / ExpressRoute + Private Endpoints.
has unknown dependenciesRun Azure Migrate dependency analysis before deciding anything.
The one-paragraph answer that demonstrates the whole skill: “I’d run Azure Migrate to inventory and dependency-map the estate, assign each workload a strategy from the 6 R’s — biased to rehost/replatform first — and land them in a governed landing zone. The web tier goes to App Service, background work to Functions, and the SQL Server to Managed Instance to preserve its surface area. I’d externalise session and files, replace config secrets with Key Vault + managed identity, secure data access over private endpoints, wire up Application Insights against an agreed SLA, codify it all in Bicep with a CI/CD pipeline, apply Hybrid Benefit and reservations for cost, and cut over in dependency-ordered waves with a tested rollback.” — That sentence touches every pillar and every section above.
