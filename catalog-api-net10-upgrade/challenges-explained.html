Upgrading a plugin-based API from .NET 6 to .NET 10

Every challenge in this migration, explained in plain language for a junior developer — including how to handle hundreds of plugin modules in batch, and the optional second act: replacing AutoMapper with a compile-time mapper. Everything here was verified by doing the upgrade for real on a production system of this shape.

1. The system we're upgrading

A layered ASP.NET Core REST API over a reference-data catalog, with a twist: its business logic is extensible through plugins.

The solution has the classic clean-architecture layers — Domain (immutable records), Application (contracts and pipeline interfaces), Persistence (EF Core + SQL Server, repositories, and the AutoMapper configuration), Processing (a pipeline that runs incoming actions through filters, saves, then notifies subscribers), and a Service host (ASP.NET Core, Serilog with file/console/SQL/email sinks, Swagger).

The twist is the module system: filters (which can inspect/veto/modify records as they flow through) and subscribers (which get notified after changes) are not compiled into the application. They are separate class libraries, deployed as folders — Modules/MyFilter/MyFilter.dll — that the host discovers and loads at runtime with Assembly.LoadFile. A production installation can have hundreds of these module DLLs, built by different people over years.

Why this matters for the upgrade: anything compiled into the solution gets checked by the compiler when we retarget it. The plugins don't — they are separate binaries that must each be retargeted, rebuilt, and redeployed, and the only thing standing between you and a runtime surprise is the process described in section 5.

2. Why upgrade — and why two separate parts

.NET 6 reached end of support in November 2024: no more security patches. .NET 10 is the current long-term-support release. This jump skips four major versions (6 → 7 → 8 → 9 → 10) in one hop — which sounds scary but mostly isn't, because Microsoft keeps runtime compatibility high; the pain concentrates in a handful of packages that changed their APIs across those years.

The work splits cleanly in two, and the split is deliberate:

Part 1  .NET 6 → .NET 10

Not optional. Retarget every project, upgrade packages, fix four known breakages, rebuild and redeploy every plugin module. AutoMapper stays exactly as it is — one battle at a time. At the end: same behavior, supported platform.

Part 2  AutoMapper → Mapwright

Optional, and safe to defer. Replace the runtime mapping engine with compile-time generated mappings. Plugins are completely unaffected — mapping is internal to two projects. Do it after Part 1 is verified green, or don't do it at all; the system runs fine either way.

Never mix the parts. If something breaks while both are in flight, you can't tell whether the framework jump or the mapping rewrite did it. Part 1 ends with a tagged, verified, shippable state; Part 2 starts from that tag.

4. Part 1, challenge by challenge

4.1 The retarget itself — easier than it looks

Changing <TargetFramework>net6.0</TargetFramework> to net10.0 in every project file is one scripted find-and-replace. The C# language version rises with it, but newer C# is almost perfectly backward-compatible — old code keeps compiling. What the jump really changes is the analysis: the .NET 10 compiler is smarter about nullability and will surface a few new warnings in old code. The playbook identifies the two known spots and the pattern for fixing any others (fix at the warned site, change nothing structural).

4.2 Packages — where the years actually show

Four .NET versions of package evolution arrive at once. Most upgrades are pure version bumps. The verified matrix in the playbook pins every package to a version combination that is known to work together — that last word matters, because Serilog sinks, EF Core, and the Extensions packages each have their own compatibility webs, and "latest of everything" is how you discover incompatibilities one restore error at a time. Two upgrades change APIs, not just versions — 4.3 and 4.4 below. Two more rules protect you from unnecessary work: AutoMapper is deliberately not upgraded (it runs fine as-is, and Part 2 may delete it anyway), and NUnit stays on 3.x (NUnit 4 renamed the whole assertion API — a rewrite disguised as a version bump).

4.3 The connection-string trap — the most dangerous item in the whole migration

This one fails at runtime only. The upgraded SQL client defaults to Encrypt=true. Every connection string that doesn't explicitly say otherwise will build perfectly and then throw SqlException: the certificate chain was issued by an authority that is not trusted the first time it touches the database.

Why: modern Microsoft.Data.SqlClient versions flipped the default — connections now demand encryption and validate the server's certificate. Most internal SQL Servers run self-signed certificates, so validation fails. The fix during migration is to append TrustServerCertificate=True; to every connection string, which keeps exactly the pre-upgrade behavior. The subtle part is finding every connection string: the app's own, but also the Serilog SQL sink's (which fails silently — logging just stops) and any hardcoded in test files. The playbook's discovery phase greps them all into a checklist precisely so none is forgotten.

The honest footnote: TrustServerCertificate=True means "encrypt but don't verify who I'm talking to." The right end-state is a properly trusted certificate on the SQL Server — that's an ops task to schedule after the migration, not a reason to block it.

4.4 The email sink — when a dependency deletes your API

The system wraps Serilog's email sink in two small helper classes so that alert emails can be configured from appsettings.json. Those helpers were built on EmailConnectionInfo — a type the sink's authors removed entirely in version 3.0 (the sink switched its internals from System.Net.Mail to MailKit). There is no "small fix": the wrapper must be rewritten against the new EmailSinkOptions API. The playbook supplies complete, drop-in replacement source for both files — including the subtle mapping of the old EnableSsl boolean onto the new connection-security enum (old true meant STARTTLS, not implicit TLS), and the rule that property names must not change because JSON configuration binds to them.

4.5 Swashbuckle — a one-line change you'd otherwise hunt for

Swagger generation (Swashbuckle 10) now depends on Microsoft.OpenApi 2.x, which moved OpenApiInfo out of the Microsoft.OpenApi.Models namespace. The compiler error looks alarming (CS0234) but the fix is deleting .Models from one using directive per affected file.

5. The plugin-module problem: hundreds of filters & subscribers

This is the area where the real installation differs most from a toy example — and where scripting turns weeks into days.

5.1 Why plugins are different

Everything inside the solution is protected by the compiler: retarget it, and every incompatibility becomes a build error you can see. Plugin modules live outside that safety net. The host finds them at startup by scanning a Modules/ directory and loading <Name>/<Name>.dll via Assembly.LoadFile, then looks inside each assembly for a class implementing the provider contract. Three consequences:

  1. No build-time check spans the boundary. A module compiled against the old contracts will load — and may fail only when first used. The fix is procedural: rebuild every module against the upgraded solution, never ship mixed builds.
  2. Type identity is fragile. The most notorious plugin failure is InvalidCastException: Unable to cast object of type 'X' to type 'X' — the same type name twice. It happens when a module's folder contains its own stale copy of a host contract DLL: .NET then has two distinct assemblies with the same types, and a cast between them fails. The rule that prevents it: module folders contain the module's own DLL and its private dependencies — never the host's contract assemblies. The playbook includes a one-liner that scans the whole deployed tree for violations.
  3. At hundreds of modules, manual anything is off the table.

5.2 The batch pipeline

The playbook treats the module fleet as a data-processing problem:

  1. Discover. Enumerate module source projects (grep for references to the Filter/Subscriber contract projects) and deployed module folders. Flag any deployed module whose source can't be found — those are your long pole, and you want to know on day one.
  2. Batch retarget. One PowerShell loop rewrites the TargetFramework in every module csproj. Idempotent — safe to rerun.
  3. Batch build with a failure report. A script builds every module and writes a CSV: module name, pass/fail, first error. In a fleet of similar modules, failures cluster — the same missing package or API change repeats across dozens. Fix one by hand, script that fix across the fleet, rebuild, repeat until the failure column is empty.
  4. Batch verify before production. A smoke test loads every DLL in the staging Modules/ tree exactly the way the host would, and asserts each one yields a provider type. Runtime loading problems get caught in CI, not in production logs.
  5. Roll out in waves. Host with zero modules first (proves the platform), then one representative filter + one subscriber (proves the seam end-to-end), then the rest in batches with the smoke test between waves.
Modules with lost source deserve honesty: if you can't find a module's source, you can (in order of preference) dig for it in old CI artifacts, rewrite it (they're typically small — one provider + one filter/subscriber class), or temporarily test whether the old .NET 6 binary happens to work on the .NET 10 host in staging. That last option is recorded technical debt with a name and an owner, not a solution.

6. Part 2: replacing AutoMapper with Mapwright

6.1 What AutoMapper does in this system today

The API keeps two parallel models per entity — an immutable domain record and a mutable EF Core entity — and converts between them constantly. One static MapperFactory (~200 lines) declares three AutoMapper maps per entity:

  • domain → entity, with an Ignore() list for audit fields (User, Created, Modified) and every navigation property;
  • entity → domain (the plain direction);
  • entity → entity (a "self-map"), used two ways by the update pipeline: to clone a record before changing it (for the audit trail), and to copy scalars in place onto the EF-tracked instance so Entity Framework sees the changes.

A runtime test, AssertConfigurationIsValid(), is the only thing checking those maps stay in sync with the models. It runs when someone runs it.

6.2 Why replace it, and why Mapwright

Mapwright does the same job at compile time: you declare each mapping as a bodyless static partial method, and a Roslyn source generator writes the implementation as plain, debuggable C# during the build. The trade is decisively favorable in this codebase:

AutoMapper today

Rename a property → the convention match silently vanishes → a column ships as null. Caught only if the validity test runs, only after the fact. Mapping logic lives inside a reflection engine — no breakpoints, no diffs. ~200 config lines to maintain.

Mapwright after

Rename a property → build warning MW0001 within seconds, promotable to an error. A stale ignore-list entry is a build error (MW0003) — config can't rot. The generated mapper is a real C# file you can read and step through. 3–5 declaration lines per entity.

The transcription itself is mechanical and — crucially for a system with many entities — batchable: every ForMember(x, opt => opt.Ignore()) in the old factory becomes a nameof(...) in a [MapIgnore(...)] attribute. Do one entity end-to-end to calibrate, then sweep the rest and let the build's MW-diagnostic list be your to-do list. You never invent a mapping decision — you transcribe the decisions the old factory already recorded, and the compiler audits the transcription.

7. The generic pipeline seam — the one genuinely tricky bit

Here's the only place Part 2 requires actual design rather than transcription. The pipeline classes are generic — Repository<TDbModel>, ActionProcessor<TDataModel, TDbModel> — and they call AutoMapper through its runtime-generic interface: _mapper.Map<TDbModel>(record) works for any type because reflection resolves it at runtime. Mapwright's methods are per-pair statics (CodeMapper.ToEntity(...)) — a generic class can't name them.

The bridge is delegate injection, and it's simpler than it sounds: the generic class stops asking for "a mapper" and starts asking for exactly the functions it needs:

// Before: one magic object that can map anything (resolved at runtime)
internal Repository(..., IMapper mapper, ...)

// After: precisely the two operations Update() actually performs (resolved at compile time)
internal Repository(...,
    Func<TDbModel, TDbModel> clone,
    Action<TDbModel, TDbModel> copyScalars, ...)

Who supplies the delegates? The factory/DI registration — the one place that already knows the concrete types, because it constructs Repository<CodeEntity> per entity. There, a method group does the job: new Repository<CodeEntity>(adapterFactory, CodeMapper.Clone, CodeMapper.CopyScalars, ...). One line edited per entity per seam — same batch rhythm as the mapper files, and the compiler lists every construction site that still has the old shape (CS7036) so nothing can be missed.

Why this design: no reflection, no new interfaces on your entities, nothing to register, trivially unit-testable (pass a lambda), Native-AOT-safe. It converts "the repository can map anything" — which was never true, it could map exactly what the factory configured — into "the repository maps exactly what it was given," which the compiler can check.

8. How the phased playbook keeps you safe

The companion markdown files are written to be executed step-by-step — by a careful junior developer or by an LLM. The safety model is worth understanding because it's what makes the "hundreds of modules / many entities" scale manageable:

  • Discovery before action. Phase 01 builds work lists — every project, every connection string, every entity, every module — from your copy of the code. Nothing later assumes a count or a name, so a copy with 3× the modules follows the identical process.
  • GATEs. Every phase ends with objectively checkable conditions (a grep that must come back empty, a build that must be green, a count that must match). You never build on sand; a failing GATE stops the line.
  • Commit per GATE. Any mistake rolls back one phase, not the whole effort.
  • Runtime checks where the compiler can't see. The connection-string trap and the plugin fleet both fail at runtime only — so the playbook ends each part with an explicit smoke checklist (service starts, reads, writes, logs to SQL, emails, loads N modules) rather than declaring victory at "build succeeded."
  • A troubleshooting table maps every error message this migration is known to produce to its cause and fix — including the 'X' to type 'X' cast trap.
